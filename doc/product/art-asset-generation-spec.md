## 1. 产品判断

当前 Cocos 测试画面中的黄圆、蓝圆、绿矩形、白矩形等内容无法被玩家理解，根因不是配色问题，而是玩法实体缺少可识别的轮廓和语义符号。

M2 Demo 的美术目标是：玩家在 1 秒内能分清 **我是谁、要捡什么、要躲什么、哪里危险、当前天气是什么**。

优先替换顺序：

1. 玩家角色：用“小明”滑雪角色替代圆形占位。
2. 收集物和道具：用金币、护盾、加速器、磁铁替代黄圆/功能色块。
3. 障碍物：用松树、雪堆、悬崖、冰面、岩石替代矩形或纯色块。
4. 地形与背景：用雪道、远山、森林边界强化“滑雪跑酷”主题。
5. UI 与反馈：用 HUD、警告态、危险态、收集/撞击特效强化可读性。

## 2. 全局美术方向

### 2.1 统一风格

- 题材：明亮雪山滑雪跑酷。
- 风格：卡通 Q 版 + 轻写实，适合微信小游戏和抖音小游戏。
- 视角：2.5D 等距正交视角，玩家沿画面右下方向滑行。
- 光源：统一左上方冷色自然光，物体右下方有柔和短阴影。
- 轮廓：所有玩法实体必须有清晰外轮廓，小尺寸下仍可读。
- 色彩：雪地以浅蓝白为底，玩家红色系，金币金色系，道具按功能色区分，危险状态黄/红高对比。
- 画面密度：背景低噪点、低细节，玩法实体高对比、强轮廓。

### 2.2 生成通用提示词

所有可交互 Sprite 默认追加：

```text
2.5D等距正交视角，明亮雪山滑雪跑酷游戏，Q版轻写实，清晰外轮廓，左上方冷色自然光，右下方柔和短阴影，透明背景PNG，小尺寸下仍清晰可读，移动游戏资产
```

背景和 UI 默认追加：

```text
横屏16:9，1920x1080设计基准，冰雪主题，蓝白清爽配色，画面干净低噪点，玩法区域清晰，适合移动端小游戏
```

### 2.3 通用反向提示词

```text
不要抽象色块，不要纯几何图形，不要照片写实，不要复杂文字，不要水印，不要logo，不要签名，不要过度暗黑，不要高噪点，不要低对比，不要模糊边缘，不要裁切主体，不要透视混乱，不要遮挡玩法主体
```

| UI 面板 | 1024x768 / 1920x1080 | 按 Canvas 缩放 | 透明 | PNG | 不要把中文文案烘焙进图 |

安全区约束：

- 横屏基准：1920x1080。
- Cocos Canvas 参考：`FIT_HEIGHT`。
- 左右安全边距：至少 40px。
- HUD 高度：建议 88-120px。
- 场景主玩法区域：不要被背景高对比物体抢焦点。

## 5. 场景包连续图片 Prompt 清单

本节把资源从“单个资产”升级为“场景包”。一个场景包由多张图组成，每张图都有独立编号和完整提示词；同一场景包内的图片必须共享视角、光照、色彩、角色比例和材质语言，保证后续导入 Cocos 后看起来像同一个世界。

### 5.1 编号规则

编号格式：

```text
SB-<阶段>-<场景编号>-<图片序号>-<资产类型>
```

示例：

```text
SB-M2-S01-001-BG
SB-M2-S01-005-PLAYER-SKI
SB-M2-S01-011-OBSTACLE-TREE
```

字段含义：

| 字段 | 含义 | 示例 |
|---|---|---|
| `SB` | SnowBattle 项目标识 | `SB` |
| `M2` / `M3` | 阶段 | `M2` 核心 Demo、`M3` 全量产品化 |
| `S01` | 场景包编号 | 初学者晴天雪道 |
| `001` | 场景内图片序号 | 从背景到前景、从通用到具体 |
| `BG` / `PLAYER` / `OBSTACLE` | 资产类型 | 背景、角色、障碍、道具、UI、FX |

### 5.2 连续性约束

同一场景包内所有图片必须遵循以下锚点：

- 摄像机：2.5D 等距正交视角，玩家主要朝画面右下方滑行。
- 光照：左上方冷色自然光，右下方柔和短阴影。
- 色彩：雪地浅蓝白，阴影淡蓝，危险态黄/红，UI 蓝白冰雪风。
- 比例：角色显示高度约等于松树高度的 70%，金币显示高度约等于角色高度的 35%。
- 材质：卡通 Q 版 + 轻写实，边缘清晰，低噪点。
- 背景：低细节、低对比；玩法实体：高对比、强轮廓。
- 文字：图片中不烘焙中文文案，动态文字由 Cocos Label 渲染。

如果生成工具支持参考图，建议工作流：

1. 先生成每个场景包的 `001-BG`。
2. 后续同场景图片都上传 `001-BG` 作为风格参考。

## 7. 生成与验收流程

### 7.1 推荐批量生成顺序

1. 先生成 M2 必需资源：小明、金币、松树、雪堆、悬崖、雪道背景、HUD。
2. 将资源导入 Cocos，替换当前圆形/矩形占位。
3. 截图做可读性验收：静止截图中必须能立即看懂实体含义。
4. 再生成道具、天气和 FX。
5. 最后扩展全角色、全章节和完整 UI。

### 7.2 连续场景包生成顺序

同一场景包的图片建议按编号顺序生成，不要跳着生成：

1. `001-BG`：先锁定场景光照、视角、色彩。
2. `002/003-TERRAIN`：生成可平铺地表和方向引导。
3. `PLAYER`：生成角色基础动作，再生成同角色其他动作。
4. `COLLECTIBLE/POWERUP`：生成收集物和道具，确保与角色不混淆。
5. `OBSTACLE`：生成障碍，确保底部命中范围清晰。
6. `FX`：生成叠加特效，确保不遮挡玩法主体。
7. `UI`：生成 HUD 和面板，确保不烘焙文案。

### 7.3 单张图片验收标准

- 缩到 64x64 后仍能分辨主体。
- 透明背景边缘干净，无白边或黑边。
- 主体没有被裁切。
- 玩法实体的命中范围能从底部轮廓大致判断。
- 金币不能再像普通黄色圆点。
- 玩家不能和金币、道具使用同样轮廓。
- 障碍不能依赖纯色矩形表达。
- 警告态、危险态作为叠加层使用，不破坏原障碍本体。
- UI 图不烘焙中文文案，文字由引擎渲染。

### 7.4 场景包连续性验收标准

- 同场景的地面、背景、角色、障碍和 UI 不像来自不同游戏。
- 同场景的光源方向一致，阴影都落向右下。
- 同一角色不同动作的服装、头盔、滑雪板、比例一致。
- 同一道具在场景拾取物和 HUD 图标中语义一致。
- 背景远景低对比，交互实体高对比。
- 叠加层 FX 能和任意同场景障碍组合，不改变障碍本体含义。
- 多张图片合成一帧静止截图时，玩家能立即分辨角色、金币、道具、障碍、天气、HUD。

### 7.5 M2 Demo 验收标准

M2 美术替换后，玩家应能在静止截图中立即分辨：

- 小明：红色滑雪角色，带滑雪板和移动方向。
- 金币：金色雪花压纹金币。
- 护盾、加速器、磁铁：不同形状、不同功能色。
- 树、雪堆、悬崖、冰面、岩石：每类障碍有独立轮廓。
- 警告/危险：黄色和红色覆盖层能表达状态。
- 天气：晴天/小雪/雾效不会遮挡玩法主体。
- HUD：时间、金币、分数、生命、暂停可读。
